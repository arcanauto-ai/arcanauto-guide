作業手順と役割分担 初級

Claude Codeを使い始めるとき、「何を人間がやって、何をClaudeに任せるか」を最初に整理することが重要です。この考え方が身につくと、作業効率が大幅に上がります。

人間がやること vs Claudeに任せられること

タスク 人間 Claude メモ
APIキー・認証設定 ✓ — セキュリティ上、人間が管理
作業の方向性・目標設定 ✓ — 「何を作るか」は人間が決める
最終確認・承認 ✓ — 公開・デプロイ前は必ず人間が確認
コード・文書の作成 — ✓ Claude が得意とする作業
ファイル操作・検索 — ✓ 大量ファイルも高速に処理
テスト実行・バグ修正 — ✓ 反復的な作業を自動化
Git操作（commit/push） — ✓ Hookで自動化も可能
外部サービスへの公開 ✓ △ 自動化可能だが承認フロー推奨

60点→100点フレームワーク

Claude Codeとの作業は「完璧なプロンプトを一発で」ではなく、「まず60点の成果物を出してもらい、会話しながら100点に近づける」というサイクルが効率的です。

1

ざっくり指示する（60点を出させる）

「○○を作って」「△△を修正して」と大まかに伝える。完璧でなくていい。

2

成果物を確認して差分を伝える（80点へ）

「ここは○○に変えて」「この部分が足りない」と具体的にフィードバック。

3

細部を詰めて完成させる（100点へ）

エッジケース・デザイン・パフォーマンスなど仕上げ作業をClaudeに依頼。

4

人間が最終確認して完了

成果物をレビューして承認。公開・保存・共有は人間が判断する。

💡 コツ: 最初から完璧を求めず、「まず動くものを出してもらう」意識を持つと、Claude Codeとのやり取りがスムーズになります。CLAUDE.md にプロジェクトのコンテキストを書いておくと、毎回説明する手間が省けます。

ローカル環境構築 初級

Claude Codeをすぐに使い始めるための、シンプルなローカルフォルダ構造を作りましょう。最初から完璧に整える必要はありません。まず動かすことが大切です。

推奨フォルダ構造（個人利用）

~/
├── projects/               # 全プロジェクトのルート
│   ├── my-app/             # 各プロジェクト
│   │   ├── CLAUDE.md       # このプロジェクト専用の指示
│   │   └── src/
│   ├── blog-site/
│   │   └── CLAUDE.md
│   └── shared/             # 複数プロジェクトで共有するファイル
│       ├── templates/      # よく使うテンプレート
│       └── snippets/       # コードスニペット集
└── .claude/                # Claude Code グローバル設定
    ├── CLAUDE.md           # 全プロジェクト共通の指示
    └── settings.json       # 動作設定

セットアップコマンド

以下をターミナルにコピーして実行すると、上記の構造が一括で作成されます。

# フォルダ作成
mkdir -p ~/projects/shared/{templates,snippets}
mkdir -p ~/.claude

# グローバル CLAUDE.md を作成
cat > ~/.claude/CLAUDE.md << 'EOF'
# 私のClaude Code設定

## 基本方針
- 日本語で回答する
- コードにはコメントをつける
- 変更前に確認を求める（大きな変更の場合）

## よく使うパス
- プロジェクト: ~/projects/
- テンプレート: ~/projects/shared/templates/
EOF

echo "セットアップ完了！"

CLAUDE.md — Claudeへの指示書

CLAUDE.md は「Claude へのメモ」です。プロジェクトを開くたびに読み込まれ、毎回同じ説明をしなくて済むようになります。

グローバル (~/.claude/CLAUDE.md)

全プロジェクト共通の設定。言語設定・個人の好み・よく使うパスなど。

プロジェクト (./CLAUDE.md)

このプロジェクト専用の設定。技術スタック・禁止事項・重要ファイルの説明など。

ポイント: CLAUDE.md が充実するほど、Claude への指示が短くなります。「毎回説明していること」を CLAUDE.md に書いていくのがコツです。

ファイル保存ガイド 初級

「どこにファイルを置けばいいか」は初心者の最大の疑問。Claude Code 関連ファイルは大きく2種類あります。

Claude Code 専用ディレクトリ ~/.claude/

ホームディレクトリ直下の ~/.claude/ が Claude Code の設定領域。普段は意識しなくてOKですが、構造を知っておくと役立ちます。

~/.claude/
├── CLAUDE.md          ← 全プロジェクト共通の指示（最重要）
├── settings.json      ← 動作設定（権限・モデル等）
└── projects/          ← 会話履歴・メモリが自動保存
    └── -home-user-myproject/
        └── memory/    ← 「覚えておいて」で保存される記憶
初心者が編集すべきは ~/.claude/CLAUDE.md だけ。「日本語で答えて」「コミット前にテストを実行して」などの永続的な指示をここに書く。
プロジェクトファイルの保存場所（推奨）
推奨: ~/projects/プロジェクト名/
~/projects/
├── my-blog/           ← ブログプロジェクト
├── work-tool/         ← 業務ツール
└── learning/          ← 学習用

ホームディレクトリに projects フォルダを作るのがシンプルでわかりやすい。

避けるべきパス
  • • Desktop/作業中/最終版2 — 深いネスト・日本語名は問題を起こしやすい
  • • ~/Desktop/ — デスクトップに直置きは散らかる
  • • OneDrive / Google Drive 内のプロジェクトコード — 同期ラグで git が壊れる可能性あり。ただし ~/.claude/ の設定ファイルはシンボリックリンク経由での Drive 連携は安全（中級章で解説）
  • • パスにスペース・日本語を含む場所 — シェルコマンドでエラーになりやすい
CLAUDE.md の2段構造を使いこなす

プロジェクトルートに置く CLAUDE.md はそのプロジェクト専用の指示書。グローバルとローカルを使い分ける。

~/projects/my-blog/
├── CLAUDE.md          ← このプロジェクト専用の指示
├── src/
└── package.json
~/.claude/CLAUDE.md
全プロジェクトに適用。「日本語で答えて」「絵文字は使わない」など自分のスタイル。
プロジェクト/CLAUDE.md
そのプロジェクト専用。「Python 3.11 使用」「テストは pytest」「DBはSQLite」など。
ファイル・フォルダ名のルール（初心者チェックリスト）
OK な名前
  • my-project
  • my_project
  • myProject
  • project-2024
  • todo-app-v2
避ける名前
  • my project（スペース）
  • マイプロジェクト（日本語）
  • my&project（記号）
  • 最終版(2)（括弧）
  • NEW FOLDER（大文字・スペース）
スペースや日本語を含むパスは、シェルコマンドやスクリプトで予期しないエラーの原因になります。英数字・ハイフン・アンダースコアのみを推奨。

よくある質問 & トラブル対処

Claude Code を使い始めると必ずぶつかる疑問をまとめました。

「Authentication failed」「Invalid API key」エラーが出る

APIキーの設定に問題があります。以下を順に確認してください:

# 1. 環境変数を確認
echo $ANTHROPIC_API_KEY

# 2. 再ログイン
claude logout
claude login

# 3. 手動でAPIキーを設定
export ANTHROPIC_API_KEY="sk-ant-..."
# ~/.bashrc または ~/.zshrc に追加して永続化
毎回「このコマンドを実行していいですか？」と聞いてくる

これはセキュリティのための正常な動作です。よく使う操作を自動許可するには settings.json の permissions.allow に追加します:

# ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(python *)",
      "Bash(pytest *)"
    ]
  }
}

または claude --dangerously-skip-permissions で全許可（信頼できる環境のみ）。

英語で返答してくる / 日本語にしたい

CLAUDE.mdに言語指定を追加するのが最も確実な方法です:

# ~/.claude/CLAUDE.md に追加
## 言語設定
常に日本語で回答してください。
コメントやコードの説明も日本語で書いてください。

または毎回「日本語で」と最後に付け加えるだけでも効果的です。

コンテキストが長くなってきた / 動作が遅くなった

会話が長くなるとコンテキスト窓が圧迫されます。以下の対策を取りましょう:

# 現在の会話を要約して圧縮
/compact

# または新しいセッションを開始
# （CLAUDE.mdで文脈が引き継がれる）
exit
claude

1つの作業が完了したら新セッションを始めるのがベストプラクティスです。

APIコストが心配 / 使いすぎていないか確認したい

使用量は Anthropic Console で確認できます。また、コスト節約のコツ:

  • • 通常作業は claude-sonnet-4-6（Opusは必要な時だけ）
  • • 繰り返し作業は claude-haiku-4-5 で節約
  • • /compact でコンテキストを定期的に圧縮
  • • 大きなファイルをまるごと貼り付けない（@参照を使う）
  • • Claude.ai Pro（$20/月）なら使い放題プランが使える
Claudeが同じミスを繰り返す / 言ったことを忘れる

繰り返してほしくないことはCLAUDE.mdに書きましょう。セッションをまたいで永続します:

# CLAUDE.md に追加する例
## やってはいけないこと
- console.log をコードに残さないこと
- コメントアウトされたコードを残さないこと
- TypeScriptでanyを使わないこと
- テストなしでコードを変更しないこと

MCP — AI機能を拡張するプラグイン

MCP（Model Context Protocol）はClaudeに外部ツールやデータソースを接続する仕組みです。GitHubやデータベース・Web検索など、あらゆるサービスをClaude Codeから直接操作できるようになります。

MCPのイメージ

Claude単体
会話・コード生成のみ
＋ MCP
外部ツールに接続
できること
• GitHub PR作成・レビュー
• Web検索・URL取得
• DB直接クエリ
• ライブラリ最新ドキュメント参照
• ブラウザ自動操作

MCPの追加方法（2通り）

方法1: コマンドで追加（簡単）
# Claude Code CLIで追加
claude mcp add context7 \
  -- npx -y @upstash/context7-mcp

# 追加済みMCPの確認
claude mcp list

# MCPの削除
claude mcp remove context7
方法2: 設定ファイルに直接書く
# ~/.claude.json に追記
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y",
        "@upstash/context7-mcp"]
    }
  }
}
前提: npxを使うMCPにはNode.js、uvxを使うMCPにはPython+uvが必要です。node -v で確認しましょう。

Google Drive連携設定 中級

Google Drive を「共有ブレイン」として使い、スキル・テンプレート・ナレッジをチーム全員が同期して利用できる構成を作ります。シンボリックリンクで Drive のフォルダと ~/.claude/ を繋ぐのがポイントです。

シンボリックリンクとは？

シンボリックリンクは「別の場所にあるフォルダへのショートカット」です。本体は Google Drive 上にありますが、Claude Code からは ~/.claude/skills/ にあるように見えます。

シンボリックリンクなし（個人利用）

~/.claude/
├── CLAUDE.md          ← 自分で書いた設定
├── skills/            ← 自分で作ったスキル
│   └── morning.md
├── templates/
└── knowledge/

ファイルの実体がすべてローカルに存在。
他のPCや他のメンバーには共有されない。

シンボリックリンクあり（チーム共有）

~/.claude/
├── CLAUDE.md          ← 個人設定（ローカル）
├── skills/  ──────────→ [Google Drive]/ClaudeCode/skills/
├── templates/ ────────→ [Google Drive]/ClaudeCode/templates/
└── knowledge/ ────────→ [Google Drive]/ClaudeCode/knowledge/

矢印先の実体は Drive 上にある。
Drive を更新すると全員に即時反映。

書き込みも Google Drive に直接反映される

シンボリックリンクは透過的です。~/.claude/skills/ に新しいスキルを保存すると、実際には Google Drive 側に書き込まれます。Google Drive が同期すると、リンクしている全メンバーの環境に即時反映されます。

  Drive/ClaudeCode/skills/ に書き込み
    ↓ Google Drive が自動同期
  チーム全員の ~/.claude/skills/ に反映

settings.json おすすめ設定

ファイル場所: ~/.claude/settings.json（グローバル） または .claude/settings.json（プロジェクト）

{

  "permissions": {
    "allow": [
      "Bash(git *)",        // git操作を自動許可
      "Bash(npm *)",        // npm操作を自動許可
      "Bash(pytest *)",     // テスト実行を自動許可
      "Read(**)",           // ファイル読み込みを自動許可
      "Edit(**)"            // ファイル編集を自動許可
    ],
    "deny": [
      "Bash(rm -rf *)"      // 危険な削除は拒否
    ]
  },
  // モデル設定
  "model": "claude-sonnet-4-5",
  // テーマ（dark / light）
  "theme": "dark"

コマンド履歴を活用

Stop Hook — 作業完了時に自動コミット＆プッシュ

Claude が作業を完了するたびに自動でコミット・プッシュします。手動操作ゼロでバックアップが完成します。

#!/bin/bash
# ~/.claude/hooks/auto-push.sh

# スキップフラグが存在する場合は何もしない
[ -f .skip-auto-push ] && exit 0

# .env や secret ファイルが staging されていたらブロック
if git diff --cached --name-only | grep -E '\.env|secret|credential' &>/dev/null; then
  echo "⚠️  機密ファイルが staging されています。自動プッシュを中止しました。"
  exit 1
fi

# 変更がなければスキップ
git diff --quiet && git diff --cached --quiet && exit 0

# コミット＆プッシュ
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
git add -A
git commit -m "auto: Claude 作業完了 [$TIMESTAMP]"
git push
echo "✅ 自動コミット＆プッシュ完了: $TIMESTAMP"

settings.json への登録:

{
  "hooks": {
    "Stop": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "bash ~/.claude/hooks/auto-push.sh",
        "timeout": 30
      }]
    }]
  }
}

スキップ方法: 自動プッシュしたくないプロジェクトでは touch .skip-auto-push を実行してください。

定期タスクの自動実行

Claude Code を対話なしで定期実行する方法は 3 つあります。用途に合わせて使い分けてください。

方法 1 — /schedule スキル（最も手軽）

Claude Code セッション内から登録できます。再起動後も保持されます。

# セッション内で実行
/schedule "毎日朝9時にコードレビューを実行" --cron "0 9 * * *"

# 登録済みスケジュールの確認
/schedule list

# 削除
/schedule delete <id>

方法 2 — cron（Linux / macOS）

claude -p フラグでプロンプトを直接渡し、非対話モードで実行します。

# crontab -e で編集
# 書式: 分 時 日 月 曜日

# 毎朝9時に今日のタスクを確認して Discord に通知
0 9 * * 1-5  cd ~/projects/myapp && claude -p "今日のGit状況を確認して日本語で要約してください" >> ~/logs/morning.log 2>&1

# 毎週月曜0時にコードレビューを実行
0 0 * * 1    cd ~/projects/myapp && claude -p "/code-review" >> ~/logs/review.log 2>&1

# 毎日深夜にバックアップ
0 2 * * *    claude -p "~/arcanauto/logs/ 以下の今日のログをバックアップして"

方法 3 — systemd timer（Linux / Raspberry Pi）

cron より細かい制御が可能。ログが journalctl で確認できます。

# /etc/systemd/system/claude-review.service
[Unit]
Description=Claude Code 自動レビュー

[Service]
Type=oneshot
User=arcanauto
WorkingDirectory=/home/arcanauto/projects/myapp
EnvironmentFile=/home/arcanauto/.env
ExecStart=/usr/bin/claude -p "/code-review"
# /etc/systemd/system/claude-review.timer
[Unit]
Description=週次コードレビュータイマー

[Timer]
OnCalendar=Mon 09:00:00   # 毎週月曜 9:00
Persistent=true            # 起動中に時刻を過ぎた場合も実行

[Install]
WantedBy=timers.target
# 有効化
sudo systemctl enable --now claude-review.timer

# 状態確認
systemctl status claude-review.timer
journalctl -u claude-review.service --since today

claude -p の主なオプション

-p "プロンプト"非対話モードで実行（必須）
--output-format json結果を JSON で受け取る
--allowedTools "Bash,Read"使用可能ツールを制限
--model claude-haiku-4-5軽量モデルで低コスト実行
--max-turns 10ターン数を制限（暴走防止）

ガイドの自動ブラッシュアップ

Claude API を使ってガイド自体を定期的に見直し、古い内容の検出・改善提案・自動更新まで行う仕組みです。

仕組みの概要

①取得

index.html を読み込む

→

②分析

Claude API で改善点を検出

→

③通知

Discord に提案を送信

brushup.py — 自動ブラッシュアップスクリプト

#!/usr/bin/env python3
# ~/projects/claude-code-guide/brushup.py
# 使い方: python3 brushup.py
#         python3 brushup.py --apply  # 改善を自動適用

import anthropic, json, sys
from pathlib import Path
from datetime import datetime
sys.path.insert(0, str(Path(__file__).parents[2] / "bot"))
import notifier

GUIDE = Path(__file__).parent / "index.html"
client = anthropic.Anthropic()

def analyze():
    html = GUIDE.read_text()
    # セクション内のテキストのみ抽出（トークン節約）
    import re
    text = re.sub(r'<[^>]+>', '', html)[:40000]

    msg = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=2000,
        messages=[{
            "role": "user",
            "content": f"""以下は Claude Code ガイドのHTMLテキストです。
改善できる点を JSON で返してください。

```
{text}
```

JSON 形式:
{{
  "issues": [
    {{
      "section": "セクション名",
      "type": "outdated|missing|unclear|typo",
      "description": "問題の説明",
      "suggestion": "改善案"
    }}
  ],
  "overall_score": 0-100,
  "summary": "全体的な評価"
}}"""
        }]
    )
    return json.loads(msg.content[0].text)

def run():
    print("ガイド分析中...")
    result = analyze()
    score = result.get("overall_score", 0)
    issues = result.get("issues", [])

    report = f"**ガイド自動レビュー結果** ({datetime.now():%Y-%m-%d})\n"
    report += f"品質スコア: {score}/100\n"
    report += f"検出された問題: {len(issues)}件\n\n"
    for i in issues[:5]:
        report += f"• [{i['type']}] {i['section']}: {i['description']}\n"
    report += f"\n{result.get('summary', '')}"

    notifier.send_alert(report, level="info")
    print(report)

    # レポートをファイルに保存
    report_dir = Path(__file__).parents[2] / "logs"
    report_dir.mkdir(exist_ok=True)
    (report_dir / f"brushup-{datetime.now():%Y-%m-%d}.json").write_text(
        json.dumps(result, ensure_ascii=False, indent=2)
    )
    print(f"✅ レポートを logs/ に保存しました")

if __name__ == "__main__":
    run()

--dangerously-skip-permissions の危険性

このオプションは全ての確認ダイアログをスキップし、Claude が無制限に操作を実行します。以下は実際に起きたインシデント例です。

事例 1: 「不要ファイルを削除して」という指示で、本番データを含む重要なディレクトリを誤って削除。バックアップなしで復旧不可能。

事例 2: テスト環境のつもりが本番の環境変数が読み込まれており、本番データベースを上書きしてしまった。

事例 3: git push --force を実行し、チームメンバーのコミットを上書き。強制プッシュ後に気づいたが履歴が消えた。

推奨: 完全に信頼できる自動化スクリプトで、かつ破壊的操作がないと確認できた場合のみ使用すること。本番環境では使わない。

→ 順番に実行
コンテキストウィンドウの管理

コンテキストの最後の20%では大規模作業を避けましょう。/compact で圧縮、Ctrl+R で使用量確認。

  • 単一ファイル編集 — コンテキスト消費少
  • ドキュメント更新 — 低感度
  • 大規模リファクタリング — 早めに着手
  • 複数ファイルにまたがるデバッグ — 注意
フック設定 上級

フックはツール実行のタイミングでシェルコマンドを自動実行します。

// ~/.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",        // Editまたは書き込み後
        "hooks": [
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && npx prettier --write $CLAUDE_TOOL_INPUT_FILE_PATH 2>/dev/null || true"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "notify-send 'Claude Code' 'セッション終了'"
          }
        ]
      }
    ]
  }
}

コンテキストを賢く管理する

長いセッションでは /compact を使ってコンテキストを圧縮。大きなタスクに移る前、デバッグ完了後にも実行しましょう。

CLAUDE.md で行動を安定させる

繰り返し指示することはCLAUDE.mdに書きましょう。「日本語でコメント」「コミット前にテスト実行」など。

ループ・スケジュール機能を活用

メモリを活用する

「これを覚えておいて」と言うと ~/.claude/projects/ にメモリが保存されます。次のセッションでも記憶が引き継がれます。

! でシェルを直接実行
